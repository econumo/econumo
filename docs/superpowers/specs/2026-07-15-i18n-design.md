# Multi-language support (i18n) ŌĆö Design

**Date:** 2026-07-15
**Status:** Approved for planning

## Goal

Let users run Econumo in their own language, selectable from the login page
onward, with the choice stored locally. Ship English + Russian in v1, with the
structure ready for more languages. Validate translation completeness in CI.

## Current state

- Frontend: i18next + react-i18next wired in `web/src/app/i18n.ts`, single
  English catalogue `web/src/locales/en-US.ts` (TS module, ~500 keys under
  legacy namespaces `blocks/elements/pages/modals/modules`), `{var}`
  interpolation. `locale()` in `web/src/lib/config.ts` already persists to
  localStorage and falls back to `navigator.language`. The Settings page has a
  "Language" menu row gated on `getLocaleOptions().length > 1` (today: 1). The
  API client already sends `Accept-Language` on every request.
- Backend: no i18n. User-visible English strings live in error/validation
  messages (frozen wire contract), the password-reset email
  (`internal/infra/mailer/reset.go`), and currency names (out of scope).
- `errs.FieldError` already has a `Code` string field (e.g. `IS_BLANK_ERROR`)
  that the envelope currently drops.

## Decisions (made during brainstorming)

1. **Localization strategy for backend messages:** error **codes + client
   rendering**. The backend emits stable machine codes; the SPA translates.
   Backend never translates errors itself.
2. **Envelope change is additive.** `errors` (field ŌåÆ `[strings]`) and
   `message` stay frozen English. New sibling keys carry codes.
3. **One shared catalogue file per language** at the repo root, used by both
   the SPA and the Go binary.
4. **All transactional emails localized** (current and future), language from
   `Accept-Language`.
5. **v1 languages: `en` + `ru`** (user can validate Russian). `de`/`es`/`fr`
   deferred; adding a language = one JSON file + one list entry.
6. **Key structure refactored** from legacy namespaces to feature-oriented
   namespaces (below).
7. **Selector UX:** compact globe dropdown, top-right on auth pages; Settings
   uses the existing "Language" row opening a selection dialog.
8. **Language stored locally only** (localStorage; no backend user field).
9. **Validation implemented as Go guard tests** (repo's guard-test pattern),
   running inside `make go-test` / the existing go-tests workflow.
10. **Initial Russian translations are AI-generated**, reviewed by the user.

## 1. Shared translation catalogue

```
locales/
Ōö£ŌöĆŌöĆ en.json      ŌåÉ reference catalogue
Ōö£ŌöĆŌöĆ ru.json
ŌööŌöĆŌöĆ embed.go     ŌåÉ package locales; //go:embed *.json (embed cannot reach
                    parent dirs, so the declaration lives here)
```

- Language codes are bare two-letter tags (`en`, `ru`) matching what
  `locale()` derives from `navigator.language`.
- The supported-language list is defined once per stack and trivially
  extendable: Go (`internal/infra/i18n`) and web (`getLocaleOptions()`), with
  the guard tests asserting a catalogue file exists per entry.
- Placeholder syntax is `{var}` in both stacks (i18next is already configured
  with `{`/`}` delimiters; the Go interpolator uses the same).
- The SPA imports the JSON files directly in `web/src/app/i18n.ts` (Vite
  native JSON imports; add a `server.fs.allow` entry for the repo root for the
  dev server). `web/src/locales/en-US.ts` is deleted.

### Key structure (refactored)

Top-level namespaces mirror `web/src/features/` plus shared/backends buckets:

```
common.*           buttons, validation, loader, select, list, dates, password
                   input, sort/confirm dialogs (today's elements/blocks)
auth.*             login / register / remind / reset pages
accounts.*         accounts + folders (pages, forms, modals)
transactions.*
budgets.*
classifications.*  categories / tags / payees (nested one level)
connections.*
currencies.*
onboarding.*
settings.*         settings menu, CSV import/export
user.*             profile, avatar, sessions & personal tokens
errors.*           backend error codes (new)
emails.*           backend email templates (new)
```

All ~500 `t()` call sites are re-keyed in the same change that converts
`en-US.ts` ŌåÆ `locales/en.json`. English values do not change ŌĆö it is a pure
rename, verified by the existing vitest suite plus guard check 3 below.

## 2. Frontend

- `getLocaleOptions()` returns `en` (`English`) and `ru` (`ąĀčāčüčüą║ąĖą╣`) with
  native labels. The Settings "Language" row auto-appears (already gated on
  `length > 1`) and opens a selection dialog.
- New `LanguageSelector` component (globe icon + current code, dropdown),
  rendered top-right on login, register, remind-password, and reset-password
  pages.
- Selecting a language: `locale(value)` (localStorage persist) +
  `i18n.changeLanguage(value)` + update `<html lang>`. Boot resolution stays:
  stored value ŌåÆ browser language ŌåÆ `en`.
- API error rendering: an `apiErrorMessage(err)` helper prefers the new code
  fields ŌĆö `t('errors.' + code, params)` ŌĆö and falls back to the raw English
  `message` when the code is absent or unknown. Feature code that shows
  `envelope.message` or field errors switches to this helper.

## 3. Backend

### Error codes (additive envelope)

- `errs.FieldError` gains `Params map[string]any`. Codes become
  catalogue-key-shaped constants in `errs` (e.g. `category.name_length`),
  registered in a package-level list that the guard test iterates. Existing
  `IS_BLANK_ERROR`-style codes migrate to the same shape (`common.is_blank`);
  the old spelling was never emitted on the wire, so nothing breaks.
- The error envelope gains additive keys, omitted when empty so untouched
  endpoints' goldens stay byte-identical:
  - `errorCodes`: field ŌåÆ `[{code, params}]`
  - `messageCode` (+ `messageParams`): for field-less errors, e.g.
    `Invalid credentials.` (401) and the 429 rate-limit message.
- `errors` and `message` remain frozen English exactly as today. Goldens for
  endpoints that gain codes are regenerated deliberately and diff-inspected.

Example:

```json
{
  "success": false,
  "message": "Category name must be 3-64 characters",
  "code": 400,
  "errors": {"name": ["Category name must be 3-64 characters"]},
  "errorCodes": {"name": [{"code": "category.name_length",
                            "params": {"min": 3, "max": 64}}]}
}
```

### Localized emails

- New `internal/infra/i18n` package: loads the embedded catalogues once at
  boot, exposes `T(lang, key, params)` with `{var}` interpolation and English
  fallback (unknown language or missing key ŌåÆ `en`).
- Middleware parses `Accept-Language` into `reqctx` (same pattern as
  `X-Timezone`), reduced to a supported two-letter tag.
- The remind-password flow passes the language to the mailer; subject/body
  come from `emails.reset.*`. Default (no header / unsupported) stays English.
- Future emails follow the same path: keys under `emails.<name>.*`.

## 4. CI validation (Go guard tests)

New `internal/test/i18ntest` package, running in `make go-test` ŌĆö every PR is
gated through the existing go-tests workflow; no new workflow file. Checks:

1. **Key parity**: every language file has exactly `en.json`'s key set; failure
   lists missing/extra keys per file.
2. **Placeholder parity**: the `{var}` set of each value matches `en.json`'s
   across languages.
3. **Frontend coverage**: every `t('...')` string literal under `web/src`
   resolves to a key in `en.json` (regex scan of `.ts`/`.tsx`; dynamic keys
   such as `t('errors.' + code)` are excluded from this check and covered by
   check 4 via their prefix).
4. **Backend coverage**: every registered error code has an `errors.*` entry
   in every language, and no orphan `errors.*` keys exist (exact two-way
   match). Likewise for `emails.*` keys referenced by the mailer.

## 5. Testing

- apiparity goldens: regenerate only endpoints that gain codes; all diffs
  additive.
- Unit tests: `internal/infra/i18n` (fallback, interpolation, unsupported
  language), mailer per-language rendering, envelope code serialization,
  Accept-Language parsing in middleware.
- Frontend: i18n setup test covers two languages; `LanguageSelector`
  (persists to localStorage, switches i18next, updates `<html lang>`);
  `apiErrorMessage` (code ŌåÆ translation, params interpolation, unknown code ŌåÆ
  raw message fallback).

## Out of scope

- Currency names stay English (`internal/currency/names.go`).
- German / Spanish / French catalogues (structure supports them; add
  `locales/<lang>.json` + one list entry per stack when wanted).
- Number/date format localization beyond what the browser provides.
- Storing the language server-side per user.
